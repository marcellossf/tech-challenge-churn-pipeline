Tech Challenge - Fase 01

Previsão de Churn

Um projeto completo para identificar clientes com risco de cancelamento e apoiar ações de retenção em telecom.

Marcello Vídeo de apresentação - até 5 minutos

Contexto

Problema de negócio

Uma operadora de telecom precisa identificar clientes com maior risco de cancelamento para priorizar ações de retenção.

O dataset escolhido foi o Telco Customer Churn, com 7.043 registros e 21 colunas na base bruta. A variável alvo é Churn.

Alvo: Churn 7.043 registros 21 colunas Classificação binária
Fala para ler

O projeto começa com uma pergunta simples: quais clientes têm mais chance de cancelar? A ideia é usar o modelo para ajudar a empresa a agir antes, com campanhas de retenção mais bem direcionadas.

Mostre: README - objetivo e dataset

Objetivo

Construir o projeto de ponta a ponta

PR-AUC Métrica principal
FN > FP Hipótese de custo
0.25 Ponto de corte escolhido

Escopo entregue

  • Análise dos dados, ML Canvas e modelos de comparação.
  • Rede neural em PyTorch com parada antecipada.
  • Registro dos experimentos, API, testes e documentação.
Fala para ler

O objetivo foi montar uma solução completa e reproduzível. A métrica principal foi PR-AUC, porque aqui o mais importante é identificar bem os clientes com risco de churn.

Mostre: estrutura do repo e README

O que foi feito

EDA e sinais de churn

Na análise inicial, eu olhei os tipos das colunas, valores ausentes, duplicados, distribuição do churn e variáveis importantes.

Foram tratados TotalCharges, que vinha como texto, e customerID, removido da modelagem por ser apenas identificador.

Sinais fortes

  • Contrato Month-to-month.
  • Internet Fiber optic.
  • Pagamento por Electronic check.
Depois abra: notebooks/01_eda_baselines.ipynb ou reports/

O que foi feito

Modelos de comparação

Antes da rede neural, eu treinei modelos de comparação: DummyClassifier, LogisticRegression e RandomForestClassifier.

O tratamento dos dados usa preenchimento de valores, padronização de variáveis numéricas, codificação das categóricas, seed fixa e validação estratificada.

Por que isso importa?

  • Cria uma comparação justa antes da rede neural.
  • Evita defender um modelo mais complexo sem referência.
  • Deixa os experimentos registrados no MLflow.
Fala para ler

Antes de defender a rede neural, eu precisava ver se ela realmente fazia sentido. Por isso esses modelos de comparação são importantes: eles mostram se a MLP trouxe algum ganho de verdade.

Depois abra: src/tech_challenge/models/train_baselines.py

O que foi feito

Rede neural em PyTorch

64 Primeira camada
32 Segunda camada
0.2 Dropout
Fala para ler

A rede neural tem duas camadas principais, usa ReLU, dropout e uma função de erro adequada para classificação binária. O treino foi feito em lotes e com parada antecipada para evitar treinar além do necessário.

Depois abra: src/tech_challenge/models/mlp.py e train_mlp.py

Resultado

Resultados e custo

Modelo PR-AUC ROC-AUC Recall Melhor custo
MLP PyTorch 0.6348 0.8429 0.9091 13.780
Logistic Regression 0.6334 0.8419 0.7059 14.380
Random Forest 0.6055 0.8171 0.6578 17.060
Fala para ler

A regressão logística foi bem forte, mas a MLP ficou um pouco melhor nas métricas principais. O que decidiu mesmo foi o custo: usando o ponto de corte 0.25, a MLP teve o menor custo esperado.

Depois abra: reports/model_comparison.md e docs/business_cost_analysis.md

Fechamento

Fechamento

O projeto entrega um fluxo completo: análise dos dados, modelos de comparação, rede neural em PyTorch, registro de experimentos, API, testes automatizados e documentação.

A escolha da MLP foi baseada não só em métrica técnica, mas também no menor custo esperado para o problema de churn.

Limitações e próximos passos

  • Custo de negócio ainda é hipótese.
  • Não houve validação temporal.
  • Próximo passo: publicar a API, monitorar e retreinar periodicamente.
